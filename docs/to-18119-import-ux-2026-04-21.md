## Executive Summary

**Problem:** The Test Case Import wizard has a 4-phase flow (Upload → Field Mapping → Value Mapping → Preview) that *exists on paper* but fails in practice. The parser breaks on any real-world customer spreadsheet that deviates from the sample format, and the wizard's own "Don't Import" / "Don't Map" controls crash the import. Customers hit silent failures with cryptic errors — forcing support tickets instead of successful migrations.

**Recommendation:** Two-phase fix. Phase 1 (2–3 wks): patch the broken controls and improve error messages — unblocks existing customers fast. Phase 2 (6–8 wks): replace rigid column-matching with LLM-assisted field detection + user-verified mapping — achieves the BrowserStack-style vision and removes the migration barrier for any file structure.

**Key insight:** DO NOT build new UX on top of the broken parser. Phase 1 fixes must land before Phase 2 is designed.

### Interaction Model Summary

The wizard has a **mandatory-action-on-every-row** model that is not communicated upfront. In Steps 2 and 3, every row in the mapping tables must have an explicit selection — there is no valid "leave blank" state. This is a reasonable design (prevents ambiguity), but the implementation fails on:
1. **Discovery** — users don't know "Don't map this field" exists until they hit the error
2. **Guidance** — the error that surfaces doesn't point to the unfilled rows
3. **Feedback** — banners persist after resolution; dropdowns show wrong display text

**Pattern observed:** Users will encounter the validation wall at Step 2, hunt for the unknown rows, eventually discover "Don't map this field," resolve it, then hit the same wall again at Step 3 with identical frustration. This is the highest-friction moment in the whole flow.

**Quick win:** Pre-populate all "Select field" / "Select Status" / "Select Priority" placeholders with "Don't map this field" as the default. Only mapped matches override it. Users see a complete valid state on arrival and only need to adjust what they want — not hunt for what they missed.

## 3. Root Problems

### Problem 1: Rigid Parser — Exact-Match-Only Column Detection

**What's wrong:** The importer assumes a specific spreadsheet structure: exact column header strings, first-row headers, single sheet, no merged cells, no extra columns. Any deviation causes a silent failure or misparse.

**Evidence:**
- [TO-18119](https://katalon.atlassian.net/browse/TO-18119) (In Progress, Kim-Andre) — "Import fails on common customer Excel structures due to rigid assumptions about spreadsheet layout. Blocks customer migration."
- [KSR-10784](https://katalon.atlassian.net/browse/KSR-10784) (Michigan State Univ) — custom spreadsheet with additional columns = failure
- [KSR-10708](https://katalon.atlassian.net/browse/KSR-10708) (Fidelity, $284K ARR) — Book1.csv fails but second.csv works; user cannot identify the difference
- [TO-17693](https://katalon.atlassian.net/browse/TO-17693) — "TC import broken in G3 + missing Epic/Story field support"

**Root cause:** The parser reads columns by expected name, not by structure inference. Customers' real spreadsheets have column names like "Test Name", "TC Title", "Test Case Description" — none of which match our expected "Test Case Name" header.

**Impact:** Every customer migration fails unless they manually reformat their spreadsheet to match our sample. This is the most common reason customers open KSR tickets about import.

---

### Problem 2: Wizard Controls That Break the Import

**What's wrong:** The wizard shows "Don't Import" (per-column) and "Don't Map" (per-value) options, but selecting either causes the import to fail rather than gracefully skip.

**Evidence:**
- [KSR-10784](https://katalon.atlassian.net/browse/KSR-10784) — 3 specific scenarios documented:
  1. "Don't Import" on certain columns → import fails
  2. Extra columns in spreadsheet → import fails
  3. "Don't Map" on Value Mapping second screen → import fails

**Root cause:** Backend import handler doesn't handle null/skip mappings — treats them as mapping errors rather than intentional exclusions. The UX implies flexibility that the backend doesn't support.

**Impact:** Users who try to customize the import (exactly what the wizard invites) end up worse off than if they'd never seen the mapping step. Destroys trust in the feature.

---

### Problem 3: Conditional Wizard — Mapping Step Isn't Always Shown

**What's wrong:** The 4-phase wizard only triggers Field Mapping and Value Mapping under certain conditions (likely: when column headers are ambiguous, or when Status column contains non-standard values). When conditions aren't met, the system skips directly to commit — users import without reviewing mapping.

**Evidence:**
- [TO-18121](https://katalon.atlassian.net/browse/TO-18121) description: "No mapping or preview before final import. Users commit blind — can't validate that column mapping is correct before data is written." — written by the team, who know the current state
- Confluence CST page (Jan 2026) shows mapping steps existing — indicating the wizard IS implemented but not consistently surfaced
- KSR-10784 mentions "a second screen would show up" — conditional phrasing

**Root cause:** The wizard flow has conditional branching that silently skips steps when the parser makes "confident" mappings. Users who don't trigger the mapping step have no opportunity to catch mismatches.

**Impact:** Silent data corruption. Test cases import but fields are wrong (e.g., Description mapped to Step Description). Users don't notice until test execution fails.

---

### Problem 4: Error Handling Designed for Developers

**What's wrong:** When import fails, error messages are technical or generic. No row-level identification, no field-specific guidance, no indication of which field is required.

**Evidence:**
- [KSR-10708](https://katalon.atlassian.net/browse/KSR-10708) (Fidelity, $284K ARR) — "error message doesn't provide much information as to the cause"; customers request asterisks on required fields
- Kim-Andre in meeting (2026-04-20 daily): sample file "doesn't work sometimes" — Kim's own team can't reliably reproduce

**Root cause:** Error messages were implemented as system-level catches, not user-facing guidance. No translation layer between internal validation errors and actionable user messaging.

**Impact:** Support ticket for every failed import. Fidelity ($284K ARR) is actively frustrated. Customers can't self-serve the most basic migration task.

---

### Problem 5: Narrow Import Schema — Doesn't Support Real Migration Patterns

**What's wrong:** The supported field set doesn't cover fields commonly used in tools customers are migrating from (TestRail, Zephyr, Jira/Xray). Epic/Story linkage and custom fields are missing. Worse, when a CSV column maps to a custom field that doesn't exist in the project, the wizard gives no guidance — it either silently drops the column or fails without explanation.

**Evidence:**
- [KSR-10352](https://katalon.atlassian.net/browse/KSR-10352) (Encova Insurance, $41K ARR) — "import process lacks Epic and Story fields for mapping"
- [TO-16328](https://katalon.atlassian.net/browse/TO-16328) — "As a QA Lead migrating from TestRail I want to map CSV columns to existing custom fields"
- [TO-17693](https://katalon.atlassian.net/browse/TO-17693) — missing Epic/Story field support

**Root cause:** Import schema was designed for TestOps-native test cases, not as a migration tool. Custom fields in TestOps exist per-project but aren't exposed in the import wizard. There's also no permission-aware fallback: users with no admin rights can't create custom fields, so any column that needs a new custom field is a silent dead end.

**Design requirement (PM decision, 2026-04-21):** Custom fields must be handled gracefully — the wizard must detect when a CSV column resolves to a custom field not yet configured in the project, then branch:
- **User has permission to manage custom fields (admin/QA lead):** Offer to create the field on-the-fly ("Create 'Sprint' as a custom field for this project")
- **User lacks permission (viewer/QA engineer):** Surface a clear pre-import blocker: "This column requires a custom field that hasn't been configured. Ask your project admin to add '[Sprint]' as a custom field before importing."

**Impact:** Enterprise customers (migrating from mature TestRail setups with custom fields) can't preserve their metadata. Migration is incomplete, manual effort required. The permission-unaware failure makes this invisible until after the import attempt.

## 4. Applicable Patterns

### Pattern: Document-Enriched AI (Pattern 5) — PRIMARY for Phase 2
**Applies to:** LLM-assisted field detection in Phase 2
**How it applies:** The customer's CSV/Excel IS the "document." LLM reads it, extracts column structure and sample values, proposes field mappings, shows confidence per field. User reviews AI's proposal, accepts/adjusts per column.
**Adaptation:** Attribution should show: `"Test Case Name" ← Column B ("TC Title") — matched by header similarity (High confidence)`. This builds trust in the mapping.

### Pattern: Review-Before-Submit (Pattern 6) — PRIMARY for Phase 2
**Applies to:** Mapping review step before committing data
**How it applies:** AI proposes mappings, user has final say. Especially critical for enum value mapping (Status: "Ready for Review" → "In Review"?). No data written until user confirms.
**Key principle:** AI proposes, human disposes. Non-negotiable for import — data loss is irreversible.

### Pattern: Progressive Disclosure (Pattern 2) — for Phase 2 wizard
**Applies to:** The 4-step wizard UX
**How it applies:** Step 1 shows the auto-detected mapping summary (headline). Step 2 lets user drill into per-column review. Step 3 shows row-level preview with issues highlighted. User can proceed at any step if satisfied.

## 5. Competitive Analysis

### BrowserStack Test Management ⭐ (PM's Reference) — Live Screenshots Analyzed

**Flow:** 3 steps — Upload → Map Fields → Preview & Confirm (value mapping is inline in step 2, not a separate step)

**Step 1 — Upload:**
- File drop zone, 10MB limit, CSV and `.feature` (BDD) formats
- Folder location picker ("Change Folder" / "Upload to Root Level")
- Configurable: CSV Separator (default: Comma), First Row (default: 1), File Encoding (default: UTF-8) — hidden under "Show More Fields" toggle
- Import inline images toggle (off by default)
- "Let AI do the heavy lifting instead" banner → cross-sells AI test generation, NOT import AI assistance

**Step 2 — Map Fields:**
- Header: "We have mapped **13 values** from CSV to Test Management fields automatically"
- **Auto-mapping is rules-based** (header string similarity on their own sample CSV format), NOT LLM. LLM-assisted mapping remains our Phase 2 differentiator.
- Three columns: YOUR CSV DATA | WILL BE MAPPED TO | VALUES FROM YOUR CSV
- Per-field value mapping: click "(Update)" on any row → opens "Map Values for '[Field]'" modal showing CSV value → system value pairs, with "Create new value" option per row
- **Mapping options dropdown** at top: "Default mapping" + "Create your own map..." — **saveable mapping templates confirmed**
- **Custom field handling:** Unrecognized CSV column (e.g., "Estimate") → maps to "Create New Field" with ⚠ "Configure Field" link. Page-level red banner: "Configure fields marked as 'Create New Field'. Or [Ignore these fields] to continue import." — graceful skip is one click.
- Folder path: "Folder" column uses "Split by greater than character `>`" to parse hierarchical paths like `Automation/Role/New Role`

**Step 3 — Preview & Confirm:**
- "N entries ready for import" count
- Expandable list of test cases with Folder, Template Type, Priority, Steps, Expected Results
- "Import Test Cases" CTA button

**Import progress modal:**
- Progress % with "Don't close/refresh" warning
- AI cross-sell inline: "Level up your workflow with AI-generated test cases. Try it while we finish importing!" — smart use of wait time

**What we can learn:**
- 3-step wizard (not 4) — Value mapping inline in Map Fields step is cleaner; avoids the "conditional wizard" problem we have
- Custom field "Ignore" option → Adaptation: Our permission-aware branch is more thoughtful, but the one-click "Ignore these fields" escape hatch is essential to copy
- Saveable mapping templates ("Create your own map...") → confirmed G7 must be Phase 2 scope
- Folder path separator config → we should handle path separators for hierarchical folders too
- AI cross-sell during progress wait → low-effort engagement opportunity

**Where we differentiate:**
- BrowserStack CSV-only; we support Excel natively (larger installed base)
- BrowserStack auto-mapping is rules-based; our Phase 2 LLM approach auto-detects any column name — genuine differentiator
- TestOps has requirement linkage (Jira/ADO IDs) — a migration value-add BrowserStack doesn't have
- Our custom field handling is permission-aware (create vs. config-first message) — more enterprise-appropriate than BrowserStack's blanket "Configure or Ignore"

## 9. Solution Options — Revised from Live Exploration (Phase 3 — Co-Design)

> **Revision 2026-04-21 — after live wizard walkthrough (Section 2b).** Three things changed:
> 1. **Highest-friction point is now named.** A mandatory-action-on-every-row model runs through Steps 2 and 3 — every row must have an explicit selection, there is no "leave it" state. Reasonable design, broken implementation: users don't discover "Don't map this field" until a vague error fires. This, more than the broken controls, is the worst UX moment.
> 2. **A quick win emerges.** Pre-populate all rows with "Don't map this field" as the default on arrival. Eliminates the hunting problem with one backend default change. Add to Phase 1.
> 3. **Excel support at risk.** Upload DOM shows `accept="text/csv"` (S1-4). If `.xlsx` is browser-rejected before reaching the server, our claimed Excel advantage over BrowserStack may not hold today. Needs engineering verification before finalizing competitive claims.

**Where BrowserStack doesn't solve it → our Phase 1 additions:**

BrowserStack's auto-mapping covers almost all their rows because it's tested against their own sample CSV. When a row is unmatched, their UI shows a warning banner and "Ignore" escape — but the fallback is one-size-fits-all, with no permission awareness on custom fields. We go further.

| Non-standard radio+checkbox in value dropdowns (S3-3, S3-4) | Inconsistent interaction model | Redesign to standard: checkbox list + "Skip this value" toggle |

**Impact:** Closes the gap with BrowserStack on the UX baseline. Meaningfully surpasses them on alias matching and custom field handling. Unblocks active KSR tickets.
**Tradeoff:** Still no LLM (intentional — validate alias coverage first). Customers with truly exotic column names not in the alias dictionary still must rename manually. Measure coverage after Phase 1 to inform Phase 2 go/no-go.

### Option B — Surpass BrowserStack with LLM (Phase 2)

**Framing:** BrowserStack's auto-mapping is rules-based string matching on their own sample CSV format. It works beautifully when the column is named exactly "Title" or "Priority." It fails silently on "TC Title", "Test Name", "Nombre del Caso", or any abbreviation. Our Phase 2 replaces rules with LLM intent inference — making TestOps the only import tool that handles any file, any naming convention, any language. No competitor (BrowserStack, TestRail, Zephyr, qTest) does this.

**Head-to-head: BrowserStack vs TestOps Phase 2:**

| Capability | BrowserStack | TestOps Phase 2 |
|-----------|--------------|-----------------|
| Field detection mechanism | Rules: string match on their sample CSV headers | LLM: intent inference — handles aliases, abbreviations, any language |
| "Name" → "Test Case Name" | ❌ Won't auto-map (header mismatch) | ✅ LLM infers the intent |
| "TC Title" → "Test Case Name" | ❌ | ✅ |
| UX on arrival at Step 2 | "We have mapped 13 values" (their sample matches their rules → most rows filled) | "AI detected your columns — review proposals" (genuinely inferred; every row has a proposal) |
| Confidence signal | ❌ Silent — no indication of certainty | ✅ Per-field: ●●● High / ●● Med / ●○○ Low |
| Unmatched columns | ⚠ banner + "Ignore" blanket escape | LLM pre-fills "Don't map this field" + explains: "No matching field found" |
| Custom field: unrecognized column | "Create New Field" + Ignore banner; no permission awareness | Permission-aware branch: admin/lead = "Create on the fly"; viewer = "Ask admin to configure first" |
| Mapping templates | Saves user-configured mapping manually | Saves AI-reviewed + user-confirmed mapping — intent is captured, not just column names |
| LLM in import flow | ❌ Zero. Their "AI" = TCG cross-sell shown during import progress wait, not import intelligence | ✅ Core mechanic. First-mover advantage. |
| Value mapping | Inline "(Update)" per value | Same inline pattern + LLM pre-suggests enum matches (P1 → High Priority) |

**What the LLM layer eliminates from Phase 1 UX:**
Phase 1 still requires users to notice unmatched rows and manually set them to "Don't map this field." The default is safe, but users with 13 columns still have to scan 13 rows. Phase 2's LLM fills all rows with proposals — the user's job becomes *reviewing* a complete proposal, not *building* a mapping from scratch. This is the qualitative UX leap over BrowserStack.

**Scope:**
- LLM reads uploaded file during Step 1 upload (background inference — no added wait for user)
- Infers field mapping per column: exact + fuzzy + intent (column names in any format, language, or abbreviation style)
- Pre-populates all rows: matched = high-confidence proposal; unmatched = "Don't map this field" with explanation
- User arrives at Step 2 with zero hunting — every row has a valid state; they confirm or adjust
- Confidence levels guide attention: High = scan and accept; Low = review carefully
- Value mapping: same inline (Update) model from Phase 1; LLM pre-suggests enum matches
- Custom field: permission-aware branch on unresolved columns (as decided 2026-04-21)
- Saveable templates: saves AI-reviewed + user-confirmed configuration for repeat imports

**Effort:** L — 6–8 weeks (Kim-Andre + AI Platform coordination)
**Tradeoff:** LLM inference latency (~3s during upload); masked by background run. AI Platform dependency.
**Validation gate before committing to Phase 2:** After Phase 1 ships, measure what percentage of real customer files are successfully handled by alias/fuzzy matching alone. If alias matching covers 80%+ of imports, the incremental value of adding LLM may not justify the AI Platform dependency and 6–8 week timeline. Phase 2 is the right direction — but size the problem first, then decide how much gun to bring.

### Option C — Hybrid (Recommended): A Now, B Next

**Phase 1 (3–4 wks):** Reach BrowserStack's baseline — fix broken controls, pre-populate safe defaults, inline value mapping, always-show wizard, preview with count. Close the gap with zero AI dependency.

**Phase 2 (6–8 wks after Phase 1 ships):** Lap BrowserStack — replace pre-populated safe defaults with LLM proposals, add confidence signals, handle any column name in any language. The Phase 1 interaction model (pre-populated rows, inline value mapping, count-aware preview) becomes the stable container the AI layer drops into.

**The sequencing argument with BrowserStack as the benchmark:**
BrowserStack's auto-mapping works because it's tested against their own sample format — they know exactly what it handles and it fails gracefully on their known edge cases. We need Phase 1 to learn our own edge cases (S1-4 Excel support, S2-1 alias gaps, S3-3 radio/checkbox confusion) before building AI around them. Otherwise Phase 2 inherits the same failure modes — just with AI-sounding error messages.
